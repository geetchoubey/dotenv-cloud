What is dotenv-cloud?

dotenv-cloud loads your .env files like any dotenv loader — but values can be references to secrets stored in a remote vault. At run time it resolves those references through external provider plugins, merges everything with clear precedence, and injects the result into your process. Secrets stay in your secret store; only references live in your repo.

# .env
APP_NAME=billing
DB_PASSWORD=aws-sm://prod/billing/db#password   # from AWS Secrets Manager
API_TOKEN=aws-ssm://prod/billing/api_token      # from SSM Parameter Store
# run any command with the resolved environment
dotenv-cloud run -- node server.js

Install

Homebrew (macOS & Linux)

brew install geetchoubey/tap/dotenv-cloud

macOS & Linux (script)

curl -fsSL https://geetchoubey.github.io/dotenv-cloud/install.sh | sh

Windows (PowerShell)

irm https://geetchoubey.github.io/dotenv-cloud/install.ps1 | iex

The scripts auto-detect your platform, download the matching release, and verify its SHA-256 before installing. Pin a version with DOTENV_CLOUD_VERSION=v0.1.0-beta.3, or choose a directory with DOTENV_CLOUD_BIN_DIR. Prefer to read first? install.sh · install.ps1.

Manual

Grab a prebuilt archive from the releases page, verify the .sha256, extract, and put dotenv-cloud on your PATH.

Quick start

# 1. detect remote references and install needed providers
dotenv-cloud init --yes

# 2. check everything resolves (no secrets printed)
dotenv-cloud validate --providers

# 3. run your app with the merged, resolved environment
dotenv-cloud run -- ./start.sh

How it works

Reference syntax

Any value may be a URI reference. The scheme selects the provider:

aws-sm://path/to/secret#field            # AWS Secrets Manager (optional JSON field)
aws-ssm://path/to/parameter           # AWS SSM Parameter Store

Precedence

Sources merge in a defined order (highest wins). The default is:

cli  >  system env  >  remote  >  .env.local  >  .env  >  defaults

Configurable per project; validate warns about risky orderings.

Configuration

# dotenv-cloud.toml
default_profile = "dev"

[profile.prod]
env_file = ".env.production"

[providers.aws]
region = "us-east-1"

[provider_registry]
url = "https://geetchoubey.github.io/dotenv-cloud/index.json"

Providers

Providers are standalone plugins distributed through this registry. The CLI downloads, verifies, and installs them on demand.

dotenv-cloud providers search                # list available providers
dotenv-cloud providers install aws           # latest; or aws@0.1.0-beta.3 to pin
dotenv-cloud providers list                  # what's installed
dotenv-cloud providers update                # update all
ProviderSchemesBackend
awsaws-sm, aws-ssmAWS Secrets Manager & SSM Parameter Store

AWS credentials are read from your environment exactly like the AWS CLI/SDK (env vars, shared config, IAM roles). dotenv-cloud never stores them.

Security & verification

  • Integrity: every downloaded archive is checked against a published SHA-256 digest. A mismatch aborts the install.
  • Authenticity: provider artifacts are signed with ed25519. The CLI verifies each signature against a built-in trusted key by default — no configuration needed. Installing an unsigned artifact requires an explicit --allow-unsigned.
  • Least exposure: resolved secret values are redacted in diagnostics and never written to disk unless you explicitly run build/export.

This index records each artifact's URL, SHA-256, and ed25519 signature. Artifacts are distributed as GitHub Release assets.

FAQ

How is this different from a normal dotenv loader?

A normal loader reads literal values from .env. dotenv-cloud additionally treats scheme://… values as references and resolves them from a remote secret store at run time, so production secrets never need to sit in your repository or .env files.

Where do the secrets actually come from?

From whichever provider backs the scheme. Today that's AWS Secrets Manager (aws-sm) and SSM Parameter Store (aws-ssm). Authentication uses your existing AWS environment — the same credentials the AWS CLI would use.

Will macOS or Windows flag the download?

Linux: no — there's no OS gate. macOS via Homebrew: no — brew doesn't quarantine its downloads. macOS via the script or a manual download: if Gatekeeper complains about an "unidentified developer," clear the quarantine flag (the install script does this for you):

xattr -d com.apple.quarantine /path/to/dotenv-cloud

Windows: the binaries aren't yet code-signed with an Authenticode certificate, so SmartScreen or Defender may warn that the publisher is unverified — especially if you launch the .exe from Explorer. This is cosmetic; choose "More info → Run anyway," or run it from a terminal. (The downloads are still SHA-256 verified by the installer.)

Do I have to trust this registry?

You trust the signing key, not the index. The CLI ships with the project's public key built in and verifies every provider signature against it. A tampered index can't make the CLI install an artifact that isn't signed by the trusted key.

How do I pin or roll back a version?

For the CLI: DOTENV_CLOUD_VERSION=v0.1.0-beta.3 before the install command. For a provider: dotenv-cloud providers install aws@0.1.0-beta.3.

Can I use my own private registry or provider?

Yes. Point [provider_registry].url at your own index.json (any https://, file://, or local path), and add your signing key via [provider_registry].public_key to have it trusted alongside the built-in key.

Does providers install work on Windows?

The Windows CLI itself works. Automated provider installation currently targets macOS and Linux (the registry indexes .tar.gz builds); Windows provider support is planned.
