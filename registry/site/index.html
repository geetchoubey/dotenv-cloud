Introduction

dotenv, extended with secure remote secret resolution.

Keep your dotenv workflow exactly as it is — your app still reads plain environment variables and you still keep a .env file. The one difference: a value can be a reference to a secret in a remote vault instead of a hard-coded plaintext secret. dotenv-cloud resolves those references at run time, merges them with your local values by clear precedence, and injects the result into your process. No SDK to import, nothing to rewrite.

Quick install

# macOS / Linux (Homebrew)
brew install geetchoubey/tap/dotenv-cloud

# macOS / Linux (script)
curl -fsSL https://geetchoubey.github.io/dotenv-cloud/install.sh | sh

# Windows (PowerShell)
irm https://geetchoubey.github.io/dotenv-cloud/install.ps1 | iex

Scripts auto-detect your platform, verify the SHA-256, and set up tab-completion. Verify with dotenv-cloud --version.

A sample .env

APP_NAME=billing
DB_PASSWORD=aws-secrets://prod/billing/db#password   # AWS Secrets Manager
API_TOKEN=aws-ssm:///prod/billing/api_token        # SSM (three slashes)
SESSION_KEY=vault://secret/billing#session_key      # HashiCorp Vault

Run your app

# resolve references and exec, with the merged environment
dotenv-cloud run -- node server.js

FAQ

How is this different from a normal dotenv loader?

A normal loader reads literal values. dotenv-cloud additionally treats scheme://… values as references and resolves them from a remote store at run time, so production secrets never sit in your repo.

Where do the secrets come from?

From the provider backing the scheme — today AWS Secrets Manager, AWS SSM Parameter Store, and HashiCorp Vault. Authentication uses your existing environment (AWS chain / VAULT_TOKEN); credentials are never stored.

Do I have to trust the registry?

You trust the signing key, not the index. The CLI ships with the project's public key and verifies every provider's ed25519 signature against it. A tampered index can't make it install an artifact that isn't signed by the trusted key.

Will macOS or Windows flag the download?

Linux: no. macOS via Homebrew: no. macOS via script/manual: clear the quarantine flag if Gatekeeper complains (the script does this for you). Windows: binaries aren't Authenticode-signed yet, so SmartScreen may warn — cosmetic; choose "Run anyway".

How do I pin a version?

CLI: DOTENV_CLOUD_VERSION=v0.1.1 before the install script. Provider: dotenv-cloud providers install aws@0.1.0.

# 1. detect remote references and install needed providers
dotenv-cloud init --yes

# 2. check everything resolves (no secrets printed)
dotenv-cloud validate --providers

# 3. run your app with the merged, resolved environment
dotenv-cloud run -- ./start.sh

Quickstart — dotenv-cloud.toml

Place it at the project root (discovered by walking up from the cwd). It's shared project config with no secrets — commit it.

default_profile = "dev"

[profile.dev]
env_file = ".env"

[providers.aws]
region = "us-east-1"

[provider_registry]
url = "https://geetchoubey.github.io/dotenv-cloud/index.json"

AWS

# Secrets Manager
DB_PASSWORD=aws-secrets://prod/db/password
API_KEY=aws-secrets://prod/app/config#api_key          # JSON field
DB_PASSWORD=aws-secrets://prod/db/password?version_stage=AWSCURRENT
CERT=aws-secrets://prod/cert?binary=base64           # binary must opt in

# SSM — parameter names are ABSOLUTE paths => three slashes
API_TOKEN=aws-ssm:///prod/app/api_token?with_decryption=true

Auth: the AWS default credential chain (env, shared config, SSO, IAM roles). region/profile via [providers.aws] — but prefer AWS_PROFILE over committing a profile (see the note below).

HashiCorp Vault

DB_PASSWORD=vault://secret/app#password       # KV v2: reads secret/data/app
WHOLE=vault://secret/app                       # no #field -> whole JSON object
PINNED=vault://secret/app?version=3#password   # KV v2 specific version

KV v2 (default) auto-inserts the data/ segment; KV v1 uses the path verbatim (set [providers.vault.kv].default_version). Auth: token from VAULT_TOKEN, address from VAULT_ADDR; TLS via VAULT_CACERT / VAULT_SKIP_VERIFY.

Security & verification

Every downloaded archive is checked against a published SHA-256 and an ed25519 signature verified against the CLI's built-in trusted key (on by default; --allow-unsigned opts out). Resolved values are redacted in diagnostics and never written to disk unless you run build/export.

Pitfalls. SSM needs three slashes (aws-ssm:///path) — the name is absolute. Don't commit [providers.aws].profile (it overrides AWS_PROFILE); let each developer set the env var. export/build output holds real secrets — gitignore it. A missing remote with resolution.missing = warn/ignore is skipped, not replaced by [defaults].

Reference — Config TOML

TOML, discovered as dotenv-cloud.toml by walking up from the cwd. Unknown keys are rejected (typos surface in validate) except the free-form [providers.*] tables. Every section is optional.

Sections & keys

[sensitive]—keys=[…], patterns=[…] to redact.

Precedence sources

cli (from --set) · system (process env) · remote (resolved references) · env.local · env · defaults. By default system outranks .env, and remote outranks defaults.

Full annotated example

default_profile = "dev"

[profile.dev]
env_file = ".env"
env_local_file = ".env.local"

[profile.prod]
env_file = ".env.production"

[defaults]
LOG_LEVEL = "info"

[precedence]
order = ["cli", "system", "remote", "env.local", "env", "defaults"]

[providers.aws]
region = "us-east-1"          # profile omitted on purpose — use AWS_PROFILE

[providers.vault]
address = "https://vault.example.com"
[providers.vault.kv]
default_version = "v2"

[provider_registry]
url = "https://geetchoubey.github.io/dotenv-cloud/index.json"
allow_unsigned = false

[resolution]
missing = "error"             # error | warn | ignore
concurrency = 8

[conflicts]
warn_on_any_shadow = false
error_on_shadow = []

[sensitive]
keys = ["DB_PASSWORD", "API_KEY"]
patterns = ["*_TOKEN", "*_SECRET"]
